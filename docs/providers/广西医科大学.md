## 合理验证码方案

### 推荐方案：WebView 人工认证

前端打开学校真实页面：

```text
https://cas.gxmu.edu.cn/lyuapServer/login
```

用户在 WebView 内自行完成：

```text
输入账号
输入密码
填写算术验证码
必要时完成短信或扫码校验
进入统一信息门户
点击 智慧教务（本科）
进入课表页面
```

认证成功后分两种情况处理：

1. 如果平台允许导出可用 Cookie 或登录态，后端用本次短期登录态抓取课表；只有后台刷新真实需要时再新增 `LoginSession` 表。
2. 如果 Cookie 受 HttpOnly、SameSite、设备指纹、WebVPN 或同源限制，WebView 内直接请求课表接口，把 `target=course` 的 raw-data 上传给后端解析。

GXMU 默认按第 2 种做：用户登录后不要急着关闭 WebView，前端先在同源页面内抓取课表 raw payload，后端解析并写入 CourseCache，返回 `canCloseWebview=true` 后再关闭。关闭后小程序课表页读取缓存。

### 后端职责

后端只做：

- 创建 `cas_webview` 登录上下文。
- 接收认证后的短期 Session 或 WebView 上传的 raw payload。
- 校验 raw payload 是否来自 `gxmu` 绑定流程。
- 解析课表并写入 CourseCache。
- raw-data 写入成功后，把绑定状态置为 `cached_only` 或 `active`。

后端不做：

- 不自动识别验证码。
- 不调用第三方打码服务。
- 不保存明文密码。
- 不承诺后台自动续期。

## Parser 输出要求

统一输出到 Course 模型：

```ts
interface Course {
  id?: string
  name: string
  teacher?: string
  classroom?: string
  weekday: number
  startSection: number
  endSection: number
  sections?: number[]
  weeks: number[]
  rawWeeks?: string
  campus?: string
  remark?: string
  source?: unknown
}
```

GXMU 验收时必须满足：

- 2025 年课程较多的学期必须完整解析。
- 每门课必须有课程名。
- 普通课程必须有 `weekday`、`startSection`、`endSection`、`weeks`。
- `weeks` 必须展开成真实周次数组，不能只保存原始文本。
- 非连续节次保留 `sections`。
- 特殊实践课、任选课或无固定时间课程不能强行填假的星期和节次，应进入 `remark` 并保留 `source`。
- 课程时间展示必须通过 GXMU 的 `SectionTime` 映射，不能使用 WTBU 或其他学校时间表。

## 停止条件

遇到以下情况应停止自动化并提示用户人工处理：

- 验证码无法由用户在 WebView 内完成。
- 登录后要求短信、企业微信、扫码或设备校验。
- WebView 无法进入智慧教务。
- Cookie 无法导出，且 WebView 内也不能请求课表接口。
- 课表只以图片或不可解析截图展示。
- 无法确认节次时间表。
