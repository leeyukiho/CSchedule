# 新增学校接入流程

这份文档只讲日常最简单的接入方式：开发者把学校适配做好并部署，管理员在后台点“启用”作为最后上线开关。

## 一句话结论

新增学校通常只需要改后端配置和云函数，不需要改前端，也不需要手动改数据库。

正确流程是：

1. 写学校 provider 元数据。
2. 写学校专用云函数。
3. 把学校配置加入 `connected-schools.json`。
4. 部署 backend 和云函数。
5. 管理员后台点“启用”。

## 需要改哪些文件

### 1. Provider 元数据

新增：

```text
backend/src/modules/providers/adapters/<providerId>.provider.ts
```

这里只写学校展示和能力信息，例如：

- 学校 ID、名称、简称。
- 支持课表、成绩、考试、个人信息哪些能力。
- 账号密码是否允许加密保存。
- 前端展示字段，例如课表字段、成绩字段。

不要在这里写登录学校、请求教务系统、解析页面的逻辑。

### 2. 注册 provider

修改：

```text
backend/src/modules/providers/providers.module.ts
```

把新 provider import 进来，并注册：

```ts
registry.register(xxxProvider)
```

### 3. 学校内置配置

修改：

```text
backend/prisma/data/connected-schools.json
```

新增一条学校配置，重点字段是：

- `id`
- `catalogCode`
- `name`
- `shortName`
- `providerId`
- `loginMode`
- `eduSystemType`
- `homepageUrl`
- `authUrl`
- `capabilities`
- `dataAccess`
- `providerConfig.cloudFunctions`
- `providerConfig.authConfig`
- `providerConfig.sectionTimes`，如果该学校课节时间需要配置

如果某个能力要走云函数，例如课表，就必须同时满足：

```json
"dataAccess": {
  "course": ["cloud_worker"]
}
```

并且：

```json
"providerConfig": {
  "cloudFunctions": {
    "course": { "url": "https://your-cloud-function-url.example.com/syncXxx" }
  }
}
```

只有配置了 `cloudFunctions.<target>.url` 的能力，才能声明 `cloud_worker`。

### 4. 学校专用云函数

新增：

```text
cloudfunctions/syncXxx/index.js
cloudfunctions/syncXxx/package.json
```

云函数负责：

- 登录学校教务系统。
- 按 `target` 抓取数据。
- 解析成项目统一格式。
- 返回标准 `cacheResults`。

支持的常用 target：

- `course`
- `score`
- `exam`
- `profile`

### 5. CloudBase 函数配置

建议修改：

```text
cloudbaserc.json
```

把新函数加入 `functions` 数组，方便统一管理。也可以不加，但部署时就要手动指定目录。

## 管理员“启用”做什么

管理员后台点击“启用”时，后端会尝试从：

```text
backend/prisma/data/connected-schools.json
```

按学校 `id` 或 `catalogCode` 找到内置配置，并补齐：

- `providerId`
- `loginMode`
- `authUrl`
- `homepageUrl`
- `eduSystemType`
- `capabilities`
- `dataAccess`
- `providerConfig`
- `verifiedAt`

所以，“启用”可以作为最后上线动作。

但它不是开发和部署动作。如果 backend 里没有这所学校的内置配置，或者云函数还没部署，单独点“启用”只会让状态变成已启用，实际同步仍然不可用。

## 部署顺序

推荐顺序：

```bash
cloudbase fn deploy syncXxx --dir cloudfunctions/syncXxx

cd backend
pnpm install
pnpm prisma generate
pnpm prisma migrate deploy
pnpm build
pm2 restart cschedule-backend
```

然后进入管理员后台，找到学校，点击“启用”。

## 是否需要 seed 数据库

日常新增学校不强制需要执行：

```bash
pnpm seed:schools
```

因为管理员“启用”时会从 `connected-schools.json` 补内置配置。

`pnpm seed:schools` 是另一种方式：提前把 `connected-schools.json` 写入数据库。它适合首次部署或批量同步学校目录，但不是管理员启用流程的必要步骤。

## 不需要做什么

通常不需要：

- 不需要改小程序前端代码。
- 不需要手动改数据库。
- 不需要写 SQL。
- 不需要新增 Prisma migration，除非数据库表结构真的变了。
- 不需要在后端 provider 里写学校登录、抓取、解析逻辑。

## 上线前检查

上线前至少确认：

1. `connected-schools.json` 是合法 JSON。
2. `providerId` 和 provider 文件里的 `id` 一致。
3. `providers.module.ts` 已注册新 provider。
4. `cloudFunctions.<target>.url` 指向已部署的 HTTP 触发器。
5. 云函数已部署成功。
6. 后端已重新部署。
7. 管理员后台点“启用”后，学校列表能看到该学校。
8. 用真实账号测试一次绑定和同步。

## 最常见问题

### 点了启用但同步失败

优先检查：

- 新 backend 是否已经部署。
- `connected-schools.json` 是否包含这所学校。
- `providerConfig.cloudFunctions.<target>.url` 是否配置了对应 HTTP 触发器。
- 云函数是否已经部署。
- HTTP 触发器是否能被后端访问。

### 前端要不要重新 build

一般不需要。

客户端学校列表和同步策略都来自后端接口。只有改了客户端代码或前端环境变量时，才需要重新 build 前端。

### 数据库要不要手动改

不要手动改。

正常走 backend 内置配置加管理员启用；需要批量同步时再考虑 `pnpm seed:schools`。
