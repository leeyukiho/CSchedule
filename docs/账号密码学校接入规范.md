# 账号密码学校接入规范

本文档用于接入“只需要账号密码即可登录”的学校。开发者按本文完成 provider、学校配置和学校专用云函数后，通常不需要修改小程序前端，也不需要手动改数据库。

## 接入边界

只接入满足以下条件的学校：

- 登录方式是普通账号密码，`loginMode` 使用 `direct_password`。
- 不需要验证码、短信、二维码、滑块、人脸、MFA 或强制 WebView 交互。
- 可以由学校专用云函数使用账号密码登录教务系统，并返回标准 `cacheResults`。
- 支持把账号密码加密保存在后端，用于首次导入和后续自动同步。

不在本文范围内：

- CAS / OAuth / WebVPN 复杂单点登录。
- 需要用户交互的验证码、二维码、短信或滑块。
- 只支持 WebView 用户登录后抓取的学校。
- 后端直接登录学校系统抓取页面。当前架构禁止把学校登录和解析逻辑写进 backend provider。

## 总体流程

```text
新增学校 provider 元数据
  -> 注册 provider
  -> 在 connected-schools.json 增加学校配置
  -> 新增并部署学校专用云函数
  -> 配置 cloudFunctions.<target>.url
  -> 部署 backend
  -> 管理后台启用学校
  -> 使用真实账号验收绑定和同步
```

## 文件清单

通常只需要改这些文件：

```text
backend/src/modules/providers/adapters/<providerId>.provider.ts
backend/src/modules/providers/providers.module.ts
backend/prisma/data/connected-schools.json
cloudfunctions/sync<School>/index.js
cloudfunctions/sync<School>/package.json
cloudbaserc.json
docs/cloudbase-functions.md
```

`cloudbaserc.json` 和 `docs/cloudbase-functions.md` 不是运行时强依赖，但建议同步维护，方便统一部署和排查。

## 命名规范

- `providerId` 使用小写短横线或小写字母数字，例如 `wtbu`、`whhxit`。
- provider 文件命名为 `<providerId>.provider.ts`。
- provider 导出变量命名为 `<providerId>Provider`，例如 `wtbuProvider`。
- 云函数命名为 `sync<School>`，使用 PascalCase 学校缩写，例如 `syncWtbu`。
- `connected-schools.json` 里的 `id` 与 `providerId` 默认保持一致，除非一套 provider 明确服务多所学校。

新增：

```text
backend/src/modules/providers/adapters/<providerId>.provider.ts
```

最小模板：

```ts
import { SchoolProvider } from '../provider.types'

export const xxxProvider: SchoolProvider = {
  id: 'xxx',
  meta: {
    id: 'xxx',
    name: '学校全称',
    shortName: '学校简称',
    providerId: 'xxx',
    loginMode: 'direct_password',
    eduSystemType: 'eams',
    status: 'enabled',
    verifiedAt: '2026-06-27T00:00:00.000Z',
    capabilities: {
      course: true,
      score: false,
      exam: false,
      profile: false,
    },
    credentialSave: {
      passwordVaultAllowed: true,
      autoSync: 'password_login',
      scheduledSyncSupported: true,
      title: '支持保存登录信息',
      notice:
        '保存教务账号密码后，可通过云函数完成首次导入和后续自动同步。账号密码会加密保存在后端，不会用于后端直接访问学校系统。',
      consentLabel: '加密保存账号密码，用于自动同步',
    },
    dataAccess: {
      course: ['cloud_worker'],
      score: [],
      exam: [],
      profile: [],
    },
    featureDisplay: {
      course: {
        title: '课表',
        kind: 'course_grid',
        itemPath: 'courses',
        itemFields: [
          { key: 'name', label: '课程', primary: true },
          { key: 'teacher', label: '教师' },
          { key: 'classroom', label: '教室', fallbackKeys: ['location'] },
          { key: 'weeks', label: '周次' },
        ],
        emptyText: '暂无课表数据',
      },
    },
  },
}
```

注意：

- `capabilities` 必须和已验收能力一致。没完成的 target 写 `false`。
- 只给已实现云函数同步的 target 配置 `['cloud_worker']`。
- provider 只写元数据和展示配置，不写登录、HTTP 请求、HTML 解析、Cookie 管理。
- `verifiedAt` 写实际验收日期，ISO 字符串格式。

如果支持成绩、考试、个人资料，补齐 `featureDisplay.score`、`featureDisplay.exam`、`featureDisplay.profile`。已有 provider 可作为字段展示参考。

```json
  "shortName": "学校简称",
  "province": "湖北省",
  "city": "武汉市",
  "aliases": ["学校全称", "学校简称", "XXX", "xxx"],
  "providerId": "xxx",
  "loginMode": "direct_password",
  "eduSystemType": "eams",
  "homepageUrl": "https://jw.example.edu.cn",
  "authUrl": "https://jw.example.edu.cn/login",
  "verifiedAt": "2026-06-27T00:00:00.000Z",
  "status": "enabled",
  "capabilities": {
    "course": true,
    "score": false,
    "exam": false,
    "profile": false
  },
  "dataAccess": {
    "course": ["cloud_worker"],
    "score": [],
    "exam": [],
    "profile": []
  },
  "providerConfig": {
    "baseUrl": "https://jw.example.edu.cn",
    "system": "EAMS",
    "cloudFunctions": {
      "course": {
        "functionName": "syncXxx",
        "url": "https://your-cloud-function-url.example.com/syncXxx"
      }
    },
    "authConfig": {
      "captchaRequired": false,
      "captchaKind": "none",
      "uiPreset": "password",
      "passwordTransform": "plain"
    },
    "sectionTimes": [
      { "section": 1, "start": "08:00", "end": "08:45" },
      { "section": 2, "start": "08:55", "end": "09:40" }
    ]
  }
}
```

字段要求：

- `providerId` 必须等于 provider 的 `id`。
- `loginMode` 固定为 `direct_password`。
- `authConfig.captchaRequired` 固定为 `false`。
- `authConfig.captchaKind` 固定为 `none`。
- `authConfig.uiPreset` 固定为 `password`。
- `authConfig.passwordTransform` 按云函数实际登录处理填写：`plain`、`rsa_public_key`、`sha1_salt` 或 `custom`。
- `dataAccess.<target>` 中出现 `cloud_worker` 时，必须存在 `providerConfig.cloudFunctions.<target>.url`。
- 一个云函数可以同时服务多个 target，但多个 target 的 `functionName` 和 `url` 必须一致，后端才会把这些 target 合并为一次同步请求。

常用 `eduSystemType`：

- `eams`
- `zf_jwglxt`
- `qiangzhi`
- `urp`
- `custom`
- `unknown`

## 云函数职责

新增目录：

```text
cloudfunctions/syncXxx/index.js
cloudfunctions/syncXxx/package.json
```

云函数负责：

- 校验后端内部密钥。
- 使用账号密码登录学校教务系统。
- 按 `targets` 抓取课表、成绩、考试或资料。
- 解析为项目标准缓存结构。
- 返回 `cacheResults`。

云函数不负责：

- 创建用户或账号。
- 写数据库。
- 创建同步任务。
- 读取 backend 数据库。
- 向日志输出账号、密码、Cookie、Token 或完整教务响应。

### 后端调用输入

后端自动同步会向云函数 HTTP URL 发送：

```json
{
  "source": "backend_auto_sync",
  "schoolId": "xxx",
  "providerId": "xxx",
  "targets": ["course"],
  "username": "学号",
  "password": "密码",
  "semesterId": "2025-2026-1"
}
```

请求头：

```text
content-type: application/json
x-cschedule-worker-secret: <CSCHEDULE_WORKER_SECRET>
```

云函数必须用环境变量 `CSCHEDULE_WORKER_SECRET` 校验该请求头。

### 云函数返回

成功返回：

```json
{
  "ok": true,
  "result": {
    "targets": ["course"],
    "cacheResults": [
      {
        "target": "course",
        "termId": "2025-2026-1",
        "cacheData": {
          "termId": "2025-2026-1",
          "courses": [],
          "terms": [],
          "sectionTimes": [],
          "meta": {
            "source": "cloud_worker"
          }
        },
        "parsedCount": 0,
        "sourceHash": "optional-source-hash",
        "syncedAt": "2026-06-27T00:00:00.000Z",
        "warnings": []
      }
    ]
  }
}
```

失败返回：

```json
{
  "ok": false,
  "errorCode": "LOGIN_FAILED",
  "errorMessage": "账号或密码错误"
}
```

推荐错误码：

- `SYNC_INPUT_INVALID`：参数缺失或格式错误。
- `WORKER_SECRET_NOT_CONFIGURED`：云函数未配置内部密钥。
- `WORKER_UNAUTHORIZED`：内部密钥不匹配。
- `PROVIDER_NOT_FOUND`：云函数收到不支持的 `providerId`。
- `TARGET_UNSUPPORTED`：请求了云函数不支持的 target。
- `LOGIN_FAILED`：账号或密码错误。
- `SCHOOL_SYSTEM_UNAVAILABLE`：学校系统不可用或超时。
- `PARSE_FAILED`：页面或接口返回结构变化，解析失败。
- `CLOUD_SYNC_EMPTY_RESULT`：登录成功但没有解析出可写入缓存的数据。

错误信息必须面向用户或管理员可读，但不能包含敏感数据。

## 首次绑定与后续同步

账号密码学校推荐走同一套云函数：

- 首次绑定：前端从后端获取短期 `clientImportToken`，直接调用云函数 HTTP URL，云函数返回 `cacheResults + cloudProof`，前端提交给后端入库。
- 后续同步：后端解密保存的账号密码，携带 `x-cschedule-worker-secret` 调用同一个云函数。

如果只实现后续同步，也必须保证首次绑定路径能拿到至少一个有效 target 的 `cacheResults`，否则账号无法完成可用性验证。

## 安全要求

- backend 和云函数必须配置相同的 `CSCHEDULE_WORKER_SECRET`。
- `CSCHEDULE_WORKER_SECRET` 只能存在于 backend 和云函数环境变量中，不能写进前端代码、文档示例真实值或仓库。
- 后端负责加密保存账号密码，依赖 `CREDENTIAL_VAULT_KEY`。
- 云函数日志禁止输出账号、密码、Cookie、Token、完整 HTML、完整 JSON 响应。
- 云函数请求学校系统时设置合理超时和重试，避免同步任务长时间挂起。
- 登录失败、解析失败和学校系统不可用要返回不同错误码，便于后端和管理员判断问题。

## 部署

部署云函数：

```bash
cloudbase fn deploy syncXxx --dir cloudfunctions/syncXxx
```

部署 backend：

```bash
cd backend
pnpm install
pnpm prisma generate
pnpm prisma migrate deploy
pnpm build
pm2 restart cschedule-backend
```

首次部署或批量同步学校目录时可执行：

```bash
cd backend
pnpm seed:schools
```

日常新增单所学校不强制执行 seed。管理后台启用学校时，backend 会尝试从 `connected-schools.json` 按学校 `id` 或 `catalogCode` 补齐内置配置。

后端自动同步链路测试：

```bash
curl -X POST "https://your-cloud-function-url.example.com/syncXxx" \
  -H "content-type: application/json" \
  -H "x-cschedule-worker-secret: $CSCHEDULE_WORKER_SECRET" \
  -d "{\"source\":\"backend_auto_sync\",\"schoolId\":\"xxx\",\"providerId\":\"xxx\",\"targets\":[\"course\"],\"username\":\"学号\",\"password\":\"密码\"}"
```

验收清单：

- provider 文件存在，`id`、`meta.id`、`meta.providerId` 一致。
- `providers.module.ts` 已注册新 provider。
- `connected-schools.json` 是合法 JSON。
- `connected-schools.json` 中 `providerId` 指向已注册 provider。
- `loginMode` 是 `direct_password`。
- `authConfig.captchaRequired` 是 `false`，`captchaKind` 是 `none`。
- 每个声明 `cloud_worker` 的 target 都配置了 `cloudFunctions.<target>.url`。
- 云函数已部署，HTTP URL 可被 backend 访问。
- backend 和云函数配置了相同的 `CSCHEDULE_WORKER_SECRET`。
- 使用真实账号完成一次绑定。
- 使用同一账号完成一次手动同步。
- 管理后台启用后，学校列表能返回正确的能力和同步策略。
- 课表页、成绩页、考试页、个人资料页只展示已声明支持的能力。

## 常见问题

### 绑定时报 `CLIENT_CACHE_RESULTS_UNTRUSTED`

首次绑定前端直调云函数时没有返回有效 `cloudProof`，或 `cloudProof.resultHash` 与 `cacheResults` 不匹配。检查云函数是否按当前项目的签名逻辑生成证明，以及 `CSCHEDULE_WORKER_SECRET` 是否一致。

### 手动同步时报 `CLOUD_SYNC_NOT_CONFIGURED`

通常是 `connected-schools.json` 里声明了 `cloud_worker`，但没有配置对应 `providerConfig.cloudFunctions.<target>.url`，或后端环境变量缺少 `CSCHEDULE_WORKER_SECRET`。

### 请求多个 target 时失败

后端要求同一次同步的多个 target 使用同一个云函数配置。如果 `course`、`score`、`exam`、`profile` 配了不同 URL，后端不会合并调用。优先让一个 `syncXxx` 函数支持多个 target。

### 同步成功但页面没有数据

检查 `cacheResults[].cacheData` 结构。课表必须放在 `cacheData.courses`，成绩、考试、资料必须放在 `cacheData.data`。同时确认 `target` 和页面能力声明一致。

### 学校启用后仍不可用

管理后台启用只负责把学校状态打开并补齐内置配置。云函数未部署、URL 未配置、backend 未部署、密钥不一致，都会导致实际同步不可用。

## 禁止事项

- 不要在 `backend/src/modules/providers/adapters/*.provider.ts` 写学校登录、抓取、解析逻辑。
- 不要恢复 backend 直连学校系统的旧模式。
- 不要把 `CSCHEDULE_WORKER_SECRET` 放进前端。
- 不要在日志中输出账号、密码、Cookie、Token。
- 不要把未验收的能力标为 `enabled` 或加入 `cloud_worker`。
- 不要手动修改数据库来接入学校，除非是在处理明确的数据修复任务。
