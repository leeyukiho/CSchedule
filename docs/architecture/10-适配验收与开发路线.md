# 适配验收与开发路线

## 新学校验收

### 登录

```text
[ ] 明确 LoginMode
[ ] 明确是否需要验证码
[ ] 明确是否需要 WebView
[ ] 明确 course 支持的 DataAccessMode
[ ] 登录失败能返回明确错误码
[ ] 日志不输出账号、密码、Cookie、Token、验证码
```

### 课表

```text
[ ] 明确课表入口或接口
[ ] 明确 raw payload 格式
[ ] 至少验证一个真实学期
[ ] 能解析课程名、教师、教室
[ ] 能解析星期、节次、周次
[ ] 能处理单双周、非连续周、特殊课程
[ ] 能写入 CourseCache
```

### 缓存

```text
[ ] 同步成功后 CourseCache 有对应 bindingId 数据
[ ] UserSchoolBinding.lastCachedAt 更新
[ ] GET /bindings/:bindingId/timetable 能读取缓存
[ ] 没有缓存时不伪造“已同步”
[ ] WebView 学校关闭前完成 course 缓存写入
```

成绩、考试、个人信息未实现前，不作为课表接入验收项。实现后统一写 `FeatureCache`。

## 学校适配文档

每所学校保留一份记录：

```text
docs/providers/{schoolId 或 学校名}.md
```

记录：

- 学校 ID 和名称。
- 教务系统类型。
- 登录入口。
- LoginMode。
- DataAccessMode。
- 是否需要验证码或 WebView。
- 课表入口、接口、参数。
- raw payload 格式。
- 节次时间。
- 已知风险。

不要写明文账号、密码、Cookie、Token。

## 开发顺序

### 阶段 1：稳定课表闭环

```text
1. 学校列表从 School 表读取
2. 用户绑定写 UserSchoolBinding
3. Provider 获取课表 raw payload
4. Parser 转 Course
5. 写 CourseCache
6. 课表页只读 CourseCache
```

### 阶段 2：修正同步状态

```text
1. SyncRecord 只记录一次同步尝试
2. success 必须以缓存写入成功为准
3. 有 binding 但无 CourseCache 时显示尚未同步
4. 已有 binding 的同步入口不能跳回重新绑定
5. WebView 完成后必须跳转课表缓存读取
```

### 阶段 3：扩展能力

```text
1. score/exam/profile 写 FeatureCache
2. 增加更细错误提示
3. 需要时接入 Redis 锁
4. 需要时接入 BullMQ 队列
5. 需要时新增 LoginSession / UserIdentity / SchoolProviderBinding
```

## 最终结论

第一版要克制：

```text
学校配置走 School
用户绑定走 UserSchoolBinding
课表展示走 CourseCache
其他能力走 FeatureCache
同步历史走 SyncRecord
读取不请求学校
同步不做无关能力
未来表只在真实需要时新增
```
