# 安全、限流与隐私

## 敏感信息

敏感信息包括：

- 学校账号、学号、姓名。
- 学校密码、验证码。
- Cookie、Token、CSRF、CAS ticket。
- 原始 HTML、原始 JSON、成绩、班级等个人教务数据。

## 第一版安全要求

- 不在日志输出账号、密码、Cookie、Token、验证码。
- 不长期保存学校明文密码。
- 生产环境默认不长期保存完整 raw payload。
- 调试 raw payload 必须加密并设置保留期。
- 用户解绑时清理绑定相关缓存和状态。
- 登录失败和同步失败不无限重试。
- 普通读取接口不请求学校系统。

## 凭据保存策略

第一版默认只保存两类东西：

```text
1. 解析后的业务缓存：CourseCache / FeatureCache
2. 绑定上的最小状态：UserSchoolBinding.authState / cacheState / sessionReusable 等摘要字段
```

当前没有独立 `LoginSession` 表，因此不要在实现或文档中假设可以长期复用 Cookie/Token。确实需要后台刷新时，先在绑定字段中保存加密的最小短期状态；如果业务复杂到需要多条 Session、过期管理和审计，再新增 `LoginSession` 表。

## 密码保存

`credentialSaveMode=password_vault` 是后续例外能力，不是第一版默认能力。

启用前必须满足：

- 管理员按学校启用。
- 用户对单个 binding 明确授权。
- 学校无验证码、短信、扫码、设备校验等强风控。
- Provider 验证失败不会触发账号锁定。
- 密码强加密、可删除、有过期时间。

不满足时，只能保存业务缓存或短期登录态摘要。

## 限流

第一版至少做同步去重和失败控制：

```text
同一 binding + target 同时只允许一个同步
失败不无限重试
学校超时保留旧缓存
用户手动同步可设置冷却时间
```

Redis、BullMQ、分布式锁是后续扩展选项，不是当前普通功能的前置条件。

## WebView raw-data

- raw-data 上传必须校验当前用户拥有 `bindingId`。
- 如有 `contextId`，必须校验它属于当前登录或同步流程。
- 后端必须确认学校支持该 `target` 和 `accessMode`。
- raw payload 不进入普通日志。
- raw payload 解析成功后写缓存，默认不长期保存原文。
