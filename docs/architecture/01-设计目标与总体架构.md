# 设计目标与总体架构

## 目标

本项目是课程表 App 的数据获取、解析、缓存和展示系统，不重做学校教务系统，也不绕过学校认证。

第一版只做稳定闭环：

```text
选择学校
  -> 绑定学校账号
  -> 获取课表原始数据
  -> 解析为统一 Course
  -> 写入数据库缓存
  -> 前端读取缓存展示
  -> 用户需要时手动同步
```

第一版不做：

- 全国高校全部可用。
- 复杂验证码破解。
- 长期免登录和自动续期。
- 大规模定时同步。
- 多套用户身份合并和账号迁移。
- 为普通功能增加多余中间层。

## 当前分层

```text
前端
  Taro / 微信小程序
  负责页面、交互、学校选择、绑定入口、同步入口。

后端 API
  NestJS Controller
  负责鉴权、参数校验、调用业务服务、返回稳定响应。

业务服务
  Users
  Schools
  Auth
  Timetable
  Sync
  Feedback

Provider
  负责学校差异：登录、抓取、解析、字段转换。

数据库
  PostgreSQL
  保存用户、学校目录、绑定关系、课程缓存、能力缓存、同步记录和反馈。
```

第一版的业务缓存就是数据库表里的快照，例如 `CourseCache` 和 `FeatureCache`。不要再为课表展示额外引入一套缓存数据库。前端读取课表时只读后端缓存，不在普通读取接口里请求学校系统。

## 数据流

读取课表：

```text
前端
  -> GET /bindings/:bindingId/timetable
  -> 后端校验 binding 归属
  -> 读取 CourseCache
  -> 返回展示数据和绑定状态
```

同步课表：

```text
前端点击同步
  -> POST /bindings/:bindingId/sync/course
  -> 后端校验 binding 归属
  -> 创建 SyncRecord
  -> Provider 获取或接收原始数据
  -> Parser 解析
  -> 写 CourseCache
  -> 更新 UserSchoolBinding.lastCachedAt 和 SyncRecord
```

如果学校必须 WebView 登录，同步只负责引导进入 WebView 或接收 WebView 上传的 raw payload；不要让后台在没有可用登录态时反复请求学校系统。

## 边界

- 前端只负责用户操作和展示，不解析学校差异。
- API 只做请求编排，不把普通读取和同步混在一起。
- Provider 只处理学校差异，不直接决定当前用户是否有权限。
- Parser 只把原始数据转成统一模型，不访问数据库、不发网络请求。
- TimetableService 只读写课表缓存，不触发学校登录。
- SyncService 只处理同步状态和一次同步流程，不承担学校适配逻辑。

## 后续可选项

以下能力只有在真实需求出现并有代码路径时再加入：

- Redis：用于登录上下文 TTL、同步锁或队列锁。
- BullMQ：用于同步任务异步化和限流。
- LoginSession 表：用于持久化可复用短期登录态。
- UserIdentity 表：用于微信、手机号、邮箱等外部身份复用。
- SchoolProviderBinding 表：用于一个学校多套 Provider 配置或灰度发布。
