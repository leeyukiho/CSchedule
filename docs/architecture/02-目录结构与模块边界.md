## 模块职责

- `providers/shared` 放通用工具。
- 每所学校一个独立目录。
- 不同学校不要共享登录实现，除非确认协议完全一致。
- 不同学校可以共享 Parser 基类或通用周次、节次解析工具。
- Provider 不直接访问用户数据库。
- 业务服务层不关心具体学校页面、字段名和教务系统实现。
- `sessions` 或 `common/redis` 如果存在，只作为临时登录上下文、锁或队列基础设施，不代表当前必须有独立 `LoginSession` 业务表。
- `sync.processor` 和 `sync.queue` 可以作为异步实现存在；普通功能文档不应要求所有同步都必须套队列。

## Provider 内部拆分

推荐每所学校至少拆为：

```text
school.provider.ts    组合登录和能力连接器
school.login.ts       登录上下文、提交、Session 校验
school.course.ts      课表 raw 抓取
school.parser.ts      raw payload 解析
school.types.ts       学校专用类型
school.constants.ts   域名、路径、节次时间、固定参数
```

如果学校支持成绩、考试、个人信息，再增加：

```text
school.score.ts
school.exam.ts
school.profile.ts
```

## 命名约定

- 学校 ID 使用稳定英文或拼音短码，例如 `wtbu`、`whhxit`、`gxmu`。
- Provider ID 与学校 ID 可相同，也可多个学校共享一个 Provider。
- 对外 API 一律使用 `schoolId` 和 `bindingId`，不要暴露内部 Provider 文件路径。
