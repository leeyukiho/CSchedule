# 学校接入信息提交规范

本文定义用户提交学校登录方式与学校接入信息的规范。目标是让用户只需要提供学校登录入口、登录方式线索和展示信息，系统即可生成对应的登录流程配置与前端展示样式。

本文只覆盖“信息提交、配置建模、登录展示、审核流转”。数据爬取规则由后续 AI Agent 模块处理，当前不实现。

## 可行性结论

该方法可行，但边界必须清晰：

- 用户可以提交学校名称、登录入口、教务系统入口、登录字段、截图、登录方式说明和学校展示信息。
- 用户不应提交可共享的学校账号密码作为公共适配材料。
- 用户个人账号密码只能用于本人绑定学校账号时的登录流程，第一版不长期保存学校密码。
- 系统可以基于用户提交的信息生成学校候选配置，先保存在 `School.config` 或 `SchoolAccessSubmission.review` 中。
- 系统可以先展示“待适配 / 研究中 / Beta / 已启用”的学校状态。
- 数据爬取规则可以由 AI Agent 辅助识别，但必须经过基础审核后才能进入 Beta 或启用状态。

因此，这套机制适合作为“学校接入申请 + 配置生成 + 管理员审核”的流程，不适合作为完全开放的用户共同维护爬虫规则系统。

## 设计目标

```text
用户提交学校信息
  -> 系统生成学校接入申请
  -> 管理端审核基础信息
  -> 生成或更新 School 候选配置
  -> 前端按配置渲染登录方式和展示样式
  -> 数据爬取规则进入后续 AI Agent 模块处理
```

目标：

- 降低新增学校的适配门槛。
- 把“学校展示”和“登录流程”配置化。
- 避免用户直接维护抓取规则。
- 保留审核入口，防止错误、钓鱼链接和敏感信息泄露。
- 与现有 `School`、`LoginMode`、`DataAccessMode` 模型保持一致。

## 用户提交内容

用户提交的是学校接入申请，不是 Provider 实现。

```ts
interface SchoolAccessSubmission {
  id: string
  submitterUserId: string
  school: SubmittedSchoolInfo
  login: SubmittedLoginInfo
  display: SubmittedDisplayInfo
  evidence: SubmittedEvidence
  requestedTargets: DataTarget[]
  status: SubmissionStatus
  review: SubmissionReview
  createdAt: string
  updatedAt: string
}
```

### 学校基础信息

```ts
interface SubmittedSchoolInfo {
  schoolName: string
  aliases?: string[]
  province?: string
  city?: string
  officialWebsite?: string
  eduSystemWebsite?: string
  eduSystemType?: EduSystemType | 'unknown'
}
```

要求：

- `schoolName` 必填。
- `eduSystemWebsite` 建议填写教务系统或统一认证入口。
- `officialWebsite` 用于辅助判断学校真实性。
- `eduSystemType` 只是用户线索，不能直接作为最终 Provider 类型。

### 登录信息

```ts
interface SubmittedLoginInfo {
  loginUrl: string
  loginModeHint?: LoginMode
  fieldHints: SubmittedLoginFieldHint[]
  captchaHint?: CaptchaHint
  webviewRequiredHint?: boolean
  successPageHint?: string
  note?: string
}

interface SubmittedLoginFieldHint {
  name?: string
  label: string
  type: 'student_id' | 'password' | 'captcha' | 'select' | 'sms' | 'qrcode' | 'unknown'
  required?: boolean
  placeholder?: string
}

interface CaptchaHint {
  required: boolean
  type?: 'image' | 'slider' | 'sms' | 'qrcode' | 'device_check' | 'unknown'
}
```

要求：

- `loginUrl` 必填，必须是学校域名或可信统一认证域名。
- `fieldHints` 只描述字段含义，不保存用户真实账号密码。
- 如果存在滑块、短信、扫码、设备校验等复杂验证，默认进入 `cas_webview` 或 `oauth_webview` 路径。
- 用户填写的 `loginModeHint` 只作为线索，最终以审核后的配置为准。

禁止提交：

- 他人账号密码。
- 明文 Cookie、Token、CAS ticket。
- 可长期复用的学校认证材料。
- 绕过验证码、短信、扫码或设备校验的方法。

### 展示信息

```ts
interface SubmittedDisplayInfo {
  logoUrl?: string
  primaryColor?: string
  secondaryColor?: string
  loginTitle?: string
  loginSubtitle?: string
  loginHelpText?: string
  portalName?: string
}
```

用途：

- 学校列表展示。
- 登录页标题和辅助说明。
- WebView 登录页提示。
- 管理端候选学校卡片。

约束：

- `logoUrl` 应来自学校官网或管理员上传后的安全资源。
- 颜色值只能影响品牌点缀，不应改变核心交互结构。
- 前端仍以系统统一设计为主，学校样式只做轻量品牌化。

### 证据材料

```ts
interface SubmittedEvidence {
  screenshots?: EvidenceFile[]
  samplePageText?: string
  officialNoticeUrl?: string
  submitterNote?: string
}

interface EvidenceFile {
  fileId: string
  type: 'login_page' | 'course_page' | 'portal_home' | 'other'
  redacted: boolean
}
```

要求：

- 截图必须遮挡学号、姓名、手机号、身份证、课表、成绩等个人信息。
- 原始 HTML、JSON、截图如需保存，必须加密并设置保留期限。
- 证据材料只用于审核，不直接进入公开展示。

## 提交流程

```text
用户选择“添加我的学校”
  -> 搜索学校目录
  -> 未找到或未启用时创建接入申请
  -> 填写学校信息、登录入口、登录字段线索、截图
  -> 系统做基础安全检查
  -> 进入 submitted
  -> 管理端审核
  -> 生成 School 候选记录
  -> 生成 School.config 候选配置
  -> 前端可展示为 researching / beta / enabled
```

## 状态机

```ts
type SubmissionStatus =
  | 'draft'
  | 'submitted'
  | 'needs_more_info'
  | 'rejected'
  | 'accepted'
  | 'converted_to_provider_candidate'
```

```ts
type ProviderStatus =
  | 'candidate'
  | 'researching'
  | 'beta'
  | 'enabled'
  | 'disabled'
  | 'deprecated'
```

建议映射：

| 申请状态 | 学校展示状态 | 含义 |
| --- | --- | --- |
| `submitted` | `catalog_only` 或 `researching` | 已收到申请，尚未适配 |
| `needs_more_info` | `catalog_only` | 信息不足，需要补充 |
| `accepted` | `researching` | 基础信息可信，进入适配研究 |
| `converted_to_provider_candidate` | `candidate` | 已生成候选 Provider 配置 |
| AI Agent 规则审核通过 | `beta` | 可小范围试用 |
| 人工验收通过 | `enabled` | 正式启用 |

## 配置生成规则

用户提交信息审核通过后，可生成候选配置：

```ts
interface SchoolProviderConfigDraft {
  schoolId: string
  providerId: string
  loginMode: LoginMode
  dataAccess: ProviderDataAccess
  capabilities: ProviderCapabilities
  featureCapabilities: FeatureCapability[]
  eduSystemType: EduSystemType
  providerConfig: Record<string, unknown>
  featureConfig: Record<DataTarget, unknown>
  status: 'candidate' | 'researching'
  sourceSubmissionId: string
}
```

生成原则：

- 能映射到现有通用 Provider 时，优先复用现有 Provider。
- 无法识别教务系统时，`providerId` 可暂设为 `unknown_candidate`，状态为 `researching`。
- 登录方式按用户线索和审核结果映射到 `LoginMode`。
- 数据能力默认只标记用户申请的 `requestedTargets`，不能直接标为全部支持。
- `dataAccess` 在 AI Agent 或人工确认前不得标为可用。

示例：

```json
{
  "schoolId": "example_university",
  "providerId": "unknown_candidate",
  "loginMode": "cas_webview",
  "dataAccess": {
    "course": [],
    "score": [],
    "exam": [],
    "profile": []
  },
  "capabilities": {
    "course": false,
    "score": false,
    "exam": false,
    "profile": false
  },
  "providerConfig": {},
  "featureConfig": {},
  "status": "researching"
}
```

## 前端登录展示规范

前端不应写死学校登录页。应根据后端返回的登录配置渲染。

```ts
interface SchoolLoginDisplayConfig {
  schoolId: string
  schoolName: string
  logoUrl?: string
  theme?: {
    primaryColor?: string
    secondaryColor?: string
  }
  loginMode: LoginMode
  fields: LoginField[]
  webview?: WebviewLoginDescriptor
  status: ProviderStatus
  helpText?: string
}
```

渲染规则：

- `direct_password`：显示系统原生账号密码表单。
- `password_captcha`：显示账号、密码、验证码表单，验证码由后端登录上下文提供。
- `cas_simple`：显示系统原生表单或跳转统一认证上下文。
- `cas_webview` / `oauth_webview`：显示学校官方页面 WebView。
- `qrcode`：显示二维码登录状态。
- `status != enabled` 时，必须显示“研究中 / Beta / 暂不可用”等状态，不允许误导用户认为已正式支持。

## 审核规则

基础审核至少检查：

```text
[ ] 学校名称真实存在
[ ] 登录入口属于学校官网、统一认证域名或可信教务域名
[ ] 截图已脱敏
[ ] 未提交明文密码、Cookie、Token
[ ] 未包含绕过验证码或风控的方法
[ ] 登录方式能归类到明确 LoginMode
[ ] 请求能力范围明确，例如 course / score / exam / profile
[ ] 已记录提交来源和审核人
```

审核通过只代表“信息可信”，不代表数据爬取规则可用。

## 安全要求

- 用户提交材料默认不公开。
- 所有原始材料必须有保留期限。
- 敏感材料必须加密保存。
- 日志不得输出账号、密码、Cookie、Token、验证码、原始 HTML。
- 个人登录凭据只允许在本人绑定流程中使用。
- 学校接入申请不得变成共享账号池。
- 不支持验证码破解、短信绕过、扫码绕过和设备风控绕过。

## 与 AI Agent 模块的边界

当前规范不实现以下能力：

- 自动识别课程表接口。
- 自动生成爬取规则。
- 自动解析 HTML、JSON、JS、Excel、PDF。
- 自动验证课程、成绩、考试数据。
- 自动将学校状态升级为 `beta` 或 `enabled`。

这些能力统一放入后续模块，详见 `12-AI-Agent爬取规则识别与审核模块.md`。
