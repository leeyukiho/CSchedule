# 登录与数据获取方式

登录方式和数据获取方式分开描述。

## LoginMode

`LoginMode` 表示用户如何完成学校认证：

```ts
type LoginMode =
  | 'direct_password'
  | 'password_captcha'
  | 'cas_simple'
  | 'cas_webview'
  | 'oauth_webview'
  | 'qrcode'
```

`direct_password` 只表示用户输入账号密码，不表示后端会直连学校系统。

## DataAccessMode

```ts
type DataAccessMode =
  | 'cloud_worker'
  | 'webview_client_fetch'
  | 'session_import'
  | 'manual_import'
```

| 模式 | 含义 |
| --- | --- |
| `cloud_worker` | 后端保存加密凭据，后端或小程序调用学校专用云函数完成学校外网访问和解析 |
| `webview_client_fetch` | 用户在 WebView 完成认证，前端上传结构化 JSON 给后端写缓存 |
| `session_import` | 预留能力，不作为默认同步路径 |
| `manual_import` | 用户上传或粘贴导出的数据 |

旧的后端直连学校模式已删除。后端不再登录学校系统或抓取学校外网数据。

## 账号密码学校

账号密码学校的首次绑定链路：

```text
小程序获取 login-context
  -> 后端返回 syncStrategy.cloudFunctions
  -> 小程序调用学校专用云函数
  -> 云函数校验账号密码并返回 cacheResults
  -> 小程序提交 AuthService.submitLogin
  -> 后端加密保存凭据并写缓存
```

后续手动同步和自动同步链路：

```text
SyncService 创建任务
  -> 后端解密 password_vault 凭据
  -> CloudCredentialSyncService 调用学校专用云函数
  -> CourseSyncService.writeCloudCacheResult 写缓存
```

## WebView 学校

复杂验证码、CAS、扫码、设备校验或强 WebView 行为的学校走 WebView：

```text
小程序打开学校官方页面
  -> 用户完成认证
  -> WebView 内获取结构化 JSON
  -> POST /account/:accountId/raw-data
  -> 后端校验 dataAccess 并写缓存
```

WebView 路径不保存账号密码，也不参与自动同步。
