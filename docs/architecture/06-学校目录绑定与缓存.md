# 学校目录、绑定与缓存

## 结论

缓存是有用的，但这里的缓存不是额外的 Redis 缓存层，而是数据库里的业务快照。

第一版不需要把缓存单独存到另一套系统。PostgreSQL 中的 `CourseCache` 和 `FeatureCache` 就是展示层读取的数据源。这样普通读取不依赖学校系统，也不会因为学校慢、验证码、维护而影响用户查看已有课表。

## 当前数据库范围

当前 Prisma schema 只保留 8 张业务表：

```text
User
School
UserSchoolBinding
CourseCache
FeatureCache
SyncRecord
FeedbackItem
SchoolAccessSubmission
```

不要在技术文档里把以下表写成当前已实现能力：

```text
UserIdentity
LoginSession
SchoolProviderBinding
AccountMigration
ScoreCache
ExamCache
ProfileCache
```

这些只有在对应功能真实实现后再加。

## School

`School` 是学校目录和 Provider 配置表。

```text
id
name
shortName
province
city
aliases
providerId
loginMode
dataAccess
featureCapabilities
eduSystemType
homepageUrl
authUrl
enabled
status
capabilities
config
verifiedAt
createdAt
updatedAt
```

全国高校名单只表示可搜索，不表示可绑定。只有 `enabled=true` 且 `status=enabled` 的学校才允许正式绑定。

## User

`User.id` 是系统内部用户主键。第一版不引入 `UserIdentity` 表时，可以由当前登录入口创建或复用用户，但后续如果要支持微信 openid、手机号、邮箱等复用，再新增 `UserIdentity`。

学校账号不能作为 App 用户主键。

## UserSchoolBinding

`UserSchoolBinding` 表示一个用户绑定的一个学校账号。

```text
id
userId
schoolId
providerId
studentNoEncrypted
displayName
status
authState
cacheState
sessionReusable
sessionRefreshable
sessionExpireAt
lastSessionValidatedAt
credentialSaveMode
lastAuthErrorCode
lastAuthErrorAt
lastLoginAt
lastCachedAt
createdAt
updatedAt
```

状态：

| 状态 | 含义 | 前端行为 |
| --- | --- | --- |
| `active` | 绑定可用，最近登录或同步成功 | 可查看缓存，可同步 |
| `need_login` | 没有可用认证，且当前操作需要重新认证 | 引导重新登录或 WebView |
| `cached_only` | 有缓存但没有可复用认证 | 可查看缓存，刷新时重新认证 |
| `disabled` | 系统禁用 | 不允许同步 |
| `unbound` | 已解绑 | 不展示为有效绑定 |

如果数据库里只有 `User`、`School`、`UserSchoolBinding` 有数据，说明用户完成了绑定，但还没有成功写入 `CourseCache`。这时个人中心只能显示学校，课表页没有可展示缓存，状态通常应是 `need_login` 或“未同步”。真正同步成功必须看到 `CourseCache` 有对应 `bindingId` 的数据。

## CourseCache

课表缓存表，第一版核心展示数据源。

```text
id
userId
bindingId
schoolId
providerId
termId
coursesJson
termsJson
sectionTimesJson
sourceHash
syncedAt
createdAt
updatedAt
```

读取课表只查 `CourseCache`。不要在 `GET /bindings/:bindingId/timetable` 里请求学校系统。

## FeatureCache

成绩、考试、个人信息先共用一张表。

```text
id
userId
bindingId
schoolId
providerId
target        score | exam | profile
termId
dataJson
metaJson
sourceHash
syncedAt
createdAt
updatedAt
```

不要同时设计 `ScoreCache`、`ExamCache`、`ProfileCache` 三张表。除非未来某个能力有明显不同的查询需求和索引需求，再拆表。

## SyncRecord

同步记录只描述一次同步尝试，不保存课表正文。

```text
id
userId
bindingId
schoolId
providerId
target
status
errorCode
errorMessage
startedAt
finishedAt
createdAt
```

同步成功后必须写入对应缓存表，并更新 `UserSchoolBinding.lastCachedAt`。只有写了缓存，前端才能认为“已同步”。

## Raw Payload

raw payload 是解析输入，不是长期业务数据。

规则：

- 生产环境默认不长期保存完整 raw payload。
- 调试保存时必须加密并设置保留期。
- 日志只记录 hash、target、contentType、大小和错误码。
- raw payload 成功解析后写入 `CourseCache` 或 `FeatureCache`。

## 普通读取

```text
GET /bindings/:bindingId/timetable
  -> 校验当前用户拥有 bindingId
  -> 查 CourseCache
  -> 返回课程、节次时间、syncedAt、bindingStatus
```

没有缓存时返回空课表和明确状态，例如：

```text
courses: []
syncedAt: null
canSync: true
bindingStatus: need_login 或 active
message: 尚未同步课表
```

不要为了“看起来已同步”在没有缓存时伪造数据。

## 成绩、考试、个人资料的存储边界

当前仍然只使用一张 `FeatureCache` 存放非课表能力，但必须按 `target` 分开写入：

```text
target=score   -> 成绩缓存，只放成绩解析结果
target=exam    -> 考试缓存，只放考试安排解析结果
target=profile -> 学校账号个人资料缓存
```

不要把个人资料混入成绩缓存，也不要为了同步课表顺手覆盖无关 target。一次学校请求如果同时拿到了多个能力，后端也要拆成多条 `FeatureCache` 记录。每条记录都带 `bindingId`、`schoolId`、`providerId`，所以同一用户绑定多所学校时天然隔离。

`UserSchoolBinding.authState.profile` 只允许作为登录/绑定摘要和资料页兜底来源，正式展示仍优先读取 `FeatureCache(target=profile)`。用户手动编辑资料时，后端只保存学校展示配置允许编辑的字段，并继续写入 `FeatureCache(target=profile)`。

前端展示差异不通过硬编码学校 ID 解决，而是读取 Feature API 返回的 `display`：

```text
GET /bindings/:bindingId/profile
  -> data: profile cache
  -> display: profile_fields

GET /bindings/:bindingId/scores
  -> data: score cache
  -> display: score_semesters 或 raw
```

这使“以学校为组扩展”保持在 Provider 和 School 配置层：新增学校通常只新增适配器、解析器和 `featureDisplay`，不改通用页面。

## 四类数据流

所有学校都走同一条平台数据流，差异只在 Provider 内部：

```text
前端统一 UI
  -> 后端统一 API
  -> 通过 binding.schoolId 找到 School
  -> 通过 School.providerId 找到 Provider
  -> Provider 按学校抓取/解析
  -> 写入 CourseCache 或 FeatureCache
  -> API 返回 data + display
  -> 前端按 display 渲染同一套 UI
```

四类能力的缓存位置：

```text
course  -> CourseCache
score   -> FeatureCache(target=score)
exam    -> FeatureCache(target=exam)
profile -> FeatureCache(target=profile)
```

后期添加学校时的工作量应集中在：

1. 注册一个 Provider 或复用同类 Provider。
2. 在 `School` 中配置 `providerId`、登录方式、能力声明、学校级参数。
3. 实现或配置该学校的 `course/score/exam/profile` 解析。
4. 配置 `featureDisplay.course/score/exam/profile`。
5. 用同一套前端页面验证展示，不新增学校专属页面。
