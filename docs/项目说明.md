# 课程表 App 项目说明

## 项目目标

本项目做课程表 App 的数据获取、解析、缓存和展示，不重做学校教务系统，也不破解学校认证。

第一版目标：

- 用户选择学校并绑定学校账号。
- 至少支持课表同步。
- 不同学校通过 Provider 适配。
- 课表解析后写入数据库缓存。
- 前端只读取缓存展示。
- 用户手动触发同步。

第一版不做：

- 全国高校全部可用。
- 复杂验证码破解。
- 长期免登录和自动续期。
- 大规模定时同步。
- 多账号身份合并。
- 为普通功能增加多余中间层。

## 当前数据库

当前 Prisma schema 只保留 8 张业务表：

```text
User
School
UserSchoolBinding
CourseCache
FeatureCache
SyncRecord
FeedbackItem
SchoolAccessSubmission
```

缓存不是多余设计。`CourseCache` 是课表页的读取来源，避免用户每次打开 App 都请求学校系统。`FeatureCache` 预留给成绩、考试、个人信息。它们都在 PostgreSQL 里，不需要第一版再单独部署一套缓存数据库。

以下表不是第一版当前能力，后续真实需要再加：

```text
LoginSession
UserIdentity
SchoolProviderBinding
AccountMigration
ScoreCache / ExamCache / ProfileCache
```

## 核心流程

绑定：

```text
选择学校
  -> 创建登录上下文
  -> 用户提交账号密码或进入 WebView
  -> 创建或更新 UserSchoolBinding
```

同步课表：

```text
用户点击同步
  -> 创建 SyncRecord
  -> Provider 获取或接收课表 raw payload
  -> Parser 转成 Course
  -> 写 CourseCache
  -> 更新 binding.lastCachedAt
```

读取课表：

```text
打开课表页
  -> GET /bindings/:bindingId/timetable
  -> 后端只读 CourseCache
  -> 返回课程和同步时间
```

没有 `CourseCache` 时，说明尚未同步成功。已有 `UserSchoolBinding` 只代表绑定存在，不代表课表已经导入。

## 模块边界

- 前端负责页面、交互、学校选择、绑定入口、同步入口。
- 后端 API 负责鉴权、参数校验、业务编排。
- Provider 负责学校登录、抓取和 raw 格式差异。
- Parser 负责 raw payload 到统一模型的转换。
- Timetable 只读写课表缓存。
- Sync 只记录和执行同步，不承担学校适配逻辑。

## 技术取舍

第一版推荐：

```text
前端: Taro + React + TypeScript
后端: NestJS + TypeScript
数据库: PostgreSQL
ORM: Prisma
```

Redis 和 BullMQ 不是普通功能的前置条件。只有当登录上下文 TTL、同步锁、异步队列、限流和重试真的需要独立基础设施时再接入。

第一版不引入 Python。Provider、Parser 和 Worker 优先使用 TypeScript，避免多一套运行环境。

## 当前问题定位口径

如果用户已经绑定学校，但个人中心显示“未同步”，通常说明：

```text
User 有数据
School 有数据
UserSchoolBinding 有数据
CourseCache 没有对应 bindingId 的数据
```

这不是缓存无用，而是导入/同步没有成功写缓存。修复功能时应检查绑定后的同步入口、raw-data 上传、parser 和 `CourseCache` 写入链路，而不是重新设计数据库。

点击同步时，如果已经存在 binding，不应该跳回“重新绑定”作为默认动作；应该按学校 `dataAccess.course` 进入后端同步、WebView 同源抓取或手动导入流程。
