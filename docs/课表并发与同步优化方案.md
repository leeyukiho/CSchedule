# 课表并发与同步优化方案

## 背景

当前问题不是单纯“国内服务器带宽太小”，而是课表查看链路仍然依赖后端实时读取。

从当前实现看：

- `frontend/src/shared/storage.ts` 只在本地保存 `accountId`、`schoolId`、`termStarts`，没有保存完整课表数据。
- `frontend/src/pages/index/index.tsx` 在 `useDidShow` 中每次进入首页都会调用 `getTimetable(id)`。
- `frontend/src/pages/schedule/index.tsx` 在 `useDidShow` 中每次进入课表页都会调用 `getTimetable(id, termId)`，部分情况下还会为了当天学期再请求一次。
- `frontend/src/shared/api/timetable.ts` 会请求 `/account/:accountId/timetable`。
- `backend/src/modules/timetable/timetable.service.ts` 每次请求都会查询账号、查询 `CourseCache`、查询所有学期缓存并合并学期列表。

所以用户“看课表”时并不是纯本地查看，而是会触发后端 API 和数据库读取。50-150 人同时打开课表，就会变成 50-150 组 API 请求、数据库查询、JSON 序列化和网络传输。

## 目标

核心目标是把链路拆成两类：

- 查看课表：本地优先、后端弱依赖、低频校验。
- 同步课表：进入队列、限流、去重、失败可重试。

最终状态应该是：大多数用户打开课表不请求后端；只有首次登录、手动刷新、缓存过期、服务端版本变更时才拉取完整课表。

## 当前瓶颈判断

### 1. 前端没有完整本地缓存

课表数据已经存在服务端数据库 `CourseCache`，但没有持久化到用户小程序本地。页面状态里的 `timetable` 只是 React state，页面重新进入或小程序冷启动后仍然要请求后端。

### 2. 页面生命周期触发频繁

Taro 的 `useDidShow` 在页面展示时触发。用户在 tab 间切换、返回页面、重新打开小程序，都可能触发拉取课表。

### 3. 课表页可能重复拉取

课表页当前逻辑会先拉默认课表，再判断当天学期。如果当天学期不是默认学期，会再拉一次指定学期。高峰期这会放大请求量。

### 4. 后端读接口不是纯单条缓存读取

`TimetableService.getTimetable` 会执行：

- 查 `studentAccount`。
- 查当前 `courseCache`。
- 查该账号所有 `courseCache` 用于合并学期。
- 合并和排序学期列表。
- 返回完整课程 JSON。

单次请求不重，但并发上来后数据库连接、CPU JSON 序列化、网络出站都会叠加。

### 5. 同步链路更重

同步课表会访问教务系统、解析 HTML/API、写数据库、绑定身份、更新账号状态。多人同时同步时，瓶颈通常不是自己的带宽，而是：

- 教务系统响应慢或限流。
- 服务端连接数和任务并发过高。
- 数据库写入和事务竞争。
- 用户重复点击导致重复任务。

## 方案总览

优先级从高到低：

1. 前端完整课表本地缓存。
2. 服务端版本校验接口或 `ETag`。
3. 页面请求去重和 TTL。
4. 同步队列、限流、任务去重。
5. 服务端热点缓存。
6. 部署层优化，而不是先买大带宽。

## 第一阶段：本地优先查看课表

### 设计

新增本地缓存结构，按账号和学期保存完整 `TimetableCacheResponse`。

建议 key：

```text
cschedule.timetable.{accountId}.default
cschedule.timetable.{accountId}.{termId}
cschedule.timetableMeta.{accountId}
```

缓存内容：

```ts
interface StoredTimetableCache {
  accountId: string
  termId: string
  data: TimetableCacheResponse
  syncedAt?: string
  cachedAt: string
  version?: string
}
```

读取策略：

1. 页面打开时，先从本地缓存读取并立即渲染。
2. 如果本地缓存不存在，才显示 loading 并请求完整课表。
3. 如果本地缓存存在但超过 TTL，后台静默检查服务端版本。
4. 服务端版本没变，不下载完整课表。
5. 服务端版本变了，下载新课表并覆盖本地缓存。

建议 TTL：

- 查看课表：6-24 小时。
- 首页今日课程：24 小时也可以接受。
- 用户手动刷新：立即跳过 TTL。

### 预期效果

如果 100 个用户只是打开课表，理想情况下 80-95 个用户直接读本地，不访问后端。服务器压力会从“每次打开都请求完整课表”降到“少量过期校验请求”。

## 第二阶段：服务端版本校验

### 方案 A：新增轻量 meta 接口

新增接口：

```http
GET /account/:accountId/timetable/meta?termId=xxx
```

返回：

```json
{
  "accountId": "xxx",
  "termId": "2025-2026-1",
  "syncedAt": "2026-06-15T08:00:00.000Z",
  "version": "sha256-or-sourceHash",
  "courseCount": 42
}
```

客户端用本地 `version/syncedAt` 对比。如果一致，不再请求完整 `/timetable`。

### 方案 B：使用 HTTP 缓存语义

`GET /account/:accountId/timetable` 返回：

```http
ETag: "sourceHash-or-cache-version"
Cache-Control: private, max-age=3600
```

客户端下次请求带：

```http
If-None-Match: "sourceHash-or-cache-version"
```

如果没变化，服务端返回：

```http
304 Not Modified
```

小程序端实现上，显式 meta 接口通常更可控，调试也更简单。

### 后端注意点

当前 `CourseCache` 已有 `sourceHash`，可以作为版本基础。但现在 `sourceHash` 包含课程内容、账号、provider、termId，不包含 `termsJson/sectionTimesJson`。如果这些也会影响展示，建议新增或计算 `cacheVersion`，覆盖完整响应相关字段。

## 第三阶段：减少重复请求

前端需要调整：

- 首页 `useDidShow` 不应每次无条件请求后端。
- 课表页 `useDidShow` 不应每次无条件请求后端。
- 同一个页面同一个 `accountId + termId` 的请求应做 in-flight 去重。
- 课表页尽量先从本地 `terms` 判断当天学期，避免默认学期和当天学期的双请求。

建议封装统一方法：

```ts
getCachedTimetable(accountId, termId, options)
```

参数：

```ts
{
  forceRefresh?: boolean
  maxAgeMs?: number
  backgroundRevalidate?: boolean
}
```

页面只调用这个方法，不再直接调用 `getTimetable`。

## 第四阶段：同步队列和限流

### 原则

查看课表和同步课表必须彻底分离：

- 查看只读本地或服务端缓存。
- 同步是重任务，必须排队。

### 任务去重

同一个账号、同一个目标、同一个学期，在已有 `pending/running` 任务时，不创建新任务，直接返回已有 job。

去重 key：

```text
sync:{accountId}:{target}:{termId || current}
```

### 并发限制

建议分层限流：

- 全局同步并发：5-20，根据服务器配置调整。
- 单学校并发：2-5，避免打爆某个教务系统。
- 单账号并发：1。
- 单 IP 或单用户手动同步频率：例如 1 分钟 1 次。

### 队列状态

`SyncRecord` 建议包含：

- `pending`
- `running`
- `succeeded`
- `failed`
- `cancelled`

前端创建同步后只拿 jobId，然后轮询：

```http
GET /sync/:jobId
```

同步完成后前端再拉一次新课表并写入本地缓存。

### 失败策略

失败不要让用户一直重试打接口。建议：

- 教务系统超时：指数退避。
- 密码错误：不自动重试。
- 验证码/会话失效：提示用户走 WebView 或重新登录。
- 学校接口维护：返回可读错误，延迟重试。

## 第五阶段：服务端缓存

如果本地缓存和版本校验完成后仍有压力，再做服务端缓存。

### Redis 缓存完整响应

key：

```text
timetable:{accountId}:{termId || default}
```

value：完整 `TimetableCacheResponse` JSON。

TTL：

- 1-24 小时。
- 同步成功后主动删除该账号相关 key。

### Redis 缓存 meta

key：

```text
timetable-meta:{accountId}:{termId || default}
```

meta 接口非常适合缓存，因为数据小、命中率高。

### 进程内 LRU

如果暂时不想引入 Redis，可以先用进程内 LRU：

- 优点：改造小。
- 缺点：多实例不共享，重启丢失。

适合作为短期过渡，不适合作为最终方案。

## 第六阶段：部署和备案取舍

### 不建议优先买国内大带宽

课表数据体积通常不大，先买大带宽只能缓解症状，不能解决架构问题。当前更应该先降低请求次数和同步并发。

### 推荐部署形态

低风险方案：

- 国内备案小服务器作为 API 入口。
- 数据库同地域部署。
- Nginx 开 gzip/brotli。
- 静态资源尽量走小程序包或 CDN。
- 后续按需增加 Redis。

如果使用境外服务器：

- 访问中国大陆用户可能受跨境链路影响，延迟和丢包不可控。
- 如果域名解析到中国内地服务器并提供 Web 服务，通常需要 ICP 备案。
- 如果域名仅解析到中国香港或海外服务器，通常不需要 ICP 备案，但仍可能涉及公安联网备案、平台审核和合规要求。
- 小程序 request 合法域名、HTTPS 证书、微信平台审核仍然需要单独满足。

备案规则会随地区和接入商执行细节变化，最终以云厂商备案系统、管局要求和微信平台审核结果为准。

参考：

- 阿里云：不同场景下的 ICP 备案说明 FAQ
- 阿里云：备案服务器检查
- 腾讯云：是否需要备案
- 阿里云：未备案域名解析至不同地区是否可以访问

## 推荐落地顺序

### P0：立即做

- 前端保存完整课表到本地。
- 首页和课表页先读本地缓存。
- `useDidShow` 加 TTL，避免每次展示都请求后端。
- 课表页消除不必要的双请求。

### P1：一周内做

- 新增 timetable meta 接口。
- 前端实现后台版本校验。
- 手动刷新才强制拉完整课表。
- 同一个账号同一个学期请求去重。

### P2：两周内做

- 接入同步队列。
- 单账号同步去重。
- 单学校同步并发限制。
- 同步失败分类和退避。

### P3：有真实压力后做

- Redis 缓存 timetable/meta。
- API 响应 gzip/brotli。
- 数据库索引检查。
- 根据监控决定是否扩容服务器或数据库。

## 验收指标

建议用以下指标判断优化是否有效：

- 打开首页，命中本地缓存时不出现全屏 loading。
- 进入课表页，命中本地缓存时 300ms 内可展示。
- 24 小时内重复打开课表，不请求完整 `/timetable`。
- 高峰期完整课表接口 QPS 明显下降。
- 同一个账号连续点击同步，只产生一个运行任务。
- 单学校同步并发不超过配置值。
- 教务系统异常时不会出现大量立即重试。

## 结论

当前 50-150 人并发瓶颈，主要原因是“看课表”仍然依赖服务端实时读取，而不是课表数据本身必须消耗大量带宽。

正确方向是：

1. 让课表查看本地优先。
2. 用版本校验代替完整拉取。
3. 把同步放入队列并限流。
4. 在这些完成后，再按监控决定是否扩容服务器或购买更大带宽。
